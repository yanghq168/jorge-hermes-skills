# 权权内容平台矩阵

权权名下多个内容账号的每日自动内容生成系统。

## 🤖 Agent 库位置

内容 Agent 的脚本存在两个地方：

| 类型 | 路径 | 说明 |
|------|------|------|
| **执行脚本** | `~/.hermes/cron/scripts/*.py` | 实际运行的 Python Agent 程序 |
| **规范文档** | `~/.hermes/skills/productivity/content-platforms/` | SKILL.md + references/ |

每个 `.py` 脚本就是一个独立的内容 Agent，通过 `cronjob(action='create', script='xxx.py')` 注册定时任务。（v1.5.1）

| 平台 | 脚本 | 定时 | 状态 |
|------|------|------|------|
| 微信公众号「围炉家常话」 | `wechat-article-daily.py` | 每天 07:00 | ▶️ 运行 |
| 小红书周末出逃计划 | `xhs-escape-weekend.py` | 每天 10:00 | ▶️ 运行（新增） |
| **全平台内容生成** | **`unified-content-daily.py`** | **每天 17:00** | ▶️ **运行中** |
| 头条号文章 | `toutiao-article-daily.py` | 20:30 | ▶️ 运行 |
| ~~小红书旅游攻略~~ | ~~xhs-travel-daily.py~~ | ~~每天 21:30~~ | ⏸️ **已停用** |

## 🏗️ 通用架构

```
cronjob (定时) 
  → Python脚本 (~/.hermes/cron/scripts/)
    → HTML邮件模板 
      → SMTP发送至 569545015@qq.com
        → deliver=origin 推送到当前对话
```

所有脚本遵循同一模式：
1. 选题库随机抽取（扩充内容）
2. HTML精美邮件模板（渐变色头部、卡片布局）
3. `from config_loader import get_mail_config` 读取邮件配置
4. 纯文本+HTML双版本邮件
5. 输出摘要供确认

### 公众号CSS特殊处理

CSS中的 `{` 和 `}` 必须双写为 `{{` 和 `}}`（因为HTML模板用Python `.format()` 渲染，CSS大括号会冲突）。操作方法：
1. 在HTML字符串中找到 `<style>` 和 `</style>` 之间的CSS块
2. 将所有 `{` → `{{`，`}` → `}}`
3. 其余HTML部分（变量占位符如 `{title}`）保持单大括号

### 公众号选题特征

- 人设：人间清醒观察家（朋友式讲故事）
- 结构：双故事对比论证法（A反例 vs B正例）
- 标题：具体年龄/数字/身份，不用"老年人""长辈"

- 故事3元素：名字+场景+对话（王大爷/老李/小强，具体场景，对话原文）

### 常见踩坑

- **f-string反斜杠**：Python 3.12禁止在f-string内使用反斜杠，先拼接中间变量再塞入f-string
- **邮件发送失败**：先检查 `python3 ~/.hermes/cron/scripts/<script>.py` 是否能单独运行成功；若端口通但 AUTH 失败（535 Login fail），可能是腾讯云 IP 被 QQ 邮箱拒绝，此时内容改由 Lark `deliver=origin` 直推，不再依赖邮件
- **话题标签遗漏**：小红书内容必须包含 `#话题`，检查是否有 hashtags 相关变量
- **敏感词遗漏**：所有内容平台脚本必须通过 `replace()` 或模板变量实现敏感词替换，不能只依赖提示词层面的"注意用词"。必须替换的词汇包括但不限于：
  | 原词 | 替换为 | 说明 |
  |------|--------|------|
  | 拉黑 | 屏蔽 | 极端情绪词，算法过滤风险 |
  | 逼 | 让 | 攻击性词汇 |
  | 滚 | 走 | 攻击性词汇 |
- **部署到远程服务器**：内容平台脚本若需部署到远程服务器（如腾讯云），服务器通常没有GitHub凭证。应使用"本地clone + rsync"方式，而非在服务器上直接 `git clone`
- **rsync超时**：大文件/跨区域传输时rsync可能超时，改用 `tar + split + scp` 分块传输（参见 `github-repo-management` 技能的"rsync Timeouts"章节）
- **GitHub仓库代码比服务器旧**：当用户说"代码不是最新的"时，先检查服务器和GitHub哪边更新。`git diff --stat` 显示"全部删除"不代表文件缺失，只是Git未跟踪。用 `diff <(git show origin/master:FILE) FILE` 比较实际内容，确认哪边更新后再决定推送还是拉取
- **强制拉取覆盖服务器**：当用户明确要求"拉取最新代码覆盖服务器"时，使用 `git reset --hard origin/master` + `git clean -fd` 强制同步，然后 `sudo nginx -t && sudo systemctl reload nginx` 重载服务，最后用 `curl` 验证所有端点返回200

### 用户质量反馈（2026-05-28）

**问题1：邮件连发被QQ邮箱拦截**
- 问题：同时发多封邮件（公众号+头条+小红书），QQ邮箱当天触发异常
- 解决：先发1封测试→用户确认OK→再发全量。不要连发3封以上

**问题2：标题有双层书名号**
- 问题：`《《竖版粉彩...》》` → `make_title()` 返回值包含《》再被外层再加一对
- 解决：标题模板直接写 `《xxx》`，不要再套

**问题3：钩子太通用，没有场景感**
- 问题：开头说"刷到这个话题"太刻意，像机器人写的
- 解决：钩子必须有具体场景+人物+动作，例如：
  - ❌ 差："刷到这个话题，我翻了很多文章"
  - ✅ 优："前天深夜，我收到一条读者留言，半天没缓过来"
  - ✅ 优："上周回老家，看到邻居王大爷坐在门口发呆，眼眶红红的"

**问题4：内容与话题词不匹配**
- 问题："竖版粉彩盛典"是审美话题，但science模板套用了通用科普逻辑
- 解决：话题分类要更细——审美/时尚/生活类 → life模板；科普/知识类 → science模板

**问题5：缺少真实故事细节**
- 问题：故事没有具体的人名、地点、数字、对话，读起来像编的
- 解决：每个故事必须有"3元素"：名字（王大爷/老李/小强）+ 具体场景（老家/公司/医院）+ 对话（"我没想到..."/"你凭什么..."）
  - ❌ 差："有人跟我说，他很迷茫"
  - ✅ 优："老李跟我说：'我在公司干了八年，年年第一，现在说我不会带团队？'"

**问题6：标题没有冲突感**
- 问题：标题平铺直叙，没有张力，点击率低
- 解决：爆款标题公式 = 冲突/对比 + 具体数字 + 情绪词
  - ❌ 差："跟着出版社编辑沉浸式改稿：看完终于明白了"
  - ✅ 优："跟着出版社编辑改稿：我发现99%的人连初稿都不会写"
  - ✅ 优："竖版粉彩盛典刷屏，但看懂的人不到1%"

**问题7：缺少配图**
- 问题：公众号文章没有实际配图，只有生图提示词
- 解决：HTML模板中需要加入实际的图位说明（封面图放哪里、正文图放哪里、结尾图放哪里），不能只给提示词
